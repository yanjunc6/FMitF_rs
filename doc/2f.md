

# Development Plan: Verification Caching & Commutativity Splitting

> **Document Status**: Draft  
> **Last Updated**: 2026-03-18  
> **Scope**: Two independent features for the commutativity verification pipeline  

---

## Feature 1: Verification Result Caching

### 1.1 Motivation

Currently, every compiler invocation re-verifies all commutativity checks from scratch, even if the underlying program has not changed. This is wasteful for iterative development workflows where users make small, localized edits. A caching layer can skip re-verification of unchanged checks, significantly reducing turnaround time.

### 1.2 Key Requirements

| Requirement | Description |
|-------------|-------------|
| **Alpha-equivalence** | Two programs that differ only in variable naming should produce the same cache key and share results |
| **File-based persistence** | Cache must survive across compiler invocations; no language server or persistent process is assumed |
| **User-controlled lifecycle** | Users decide when to clear the cache; configurable storage limits with automatic eviction |
| **Thread safety** | Multiple verification threads may read/write the cache concurrently |
| **Soundness** | A cached "verified" result must be trustworthy — cache must be invalidated when relevant context changes |
| **Loging** | Log them in `compiler.log` |

### 1.3 Approach Overview

The caching pipeline has three stages:

```
CommutativeUnit + CfgProgram
        │
        ▼
  ┌─────────────────┐
  │ Canonicalization │   Normalize variable/table names to achieve
  │                  │   alpha-equivalence. Produce a canonical
  │                  │   representation that is identical for programs
  │                  │   differing only in naming.
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │  Fingerprinting  │   Hash the canonical representation to produce
  │                  │   a fixed-size key. This key uniquely identifies
  │                  │   the verification problem (up to alpha-equivalence).
  └────────┬────────┘
           │
           ▼
  ┌─────────────────┐
  │   Cache Lookup   │   Check file-based store for a matching entry.
  │   / Store        │   On miss, run verification and store the result.
  └─────────────────┘
```

### 1.4 Canonicalization for Alpha-Equivalence

**What this means**: Two commutativity checks that are structurally identical but use different variable names (e.g., `x` vs `tmp_0`, or `table_A` vs `table_B`) should be recognized as "the same problem." This avoids cache misses caused by cosmetic renaming.

**Approach**: Traverse the program representation (at the CFG or unit level, depending on what is most natural in the codebase) in a deterministic order. Assign canonical names to variables and tables based on the order in which they are first encountered during this traversal. For example, the first variable encountered becomes `v0`, the second `v1`, and so on. The traversal order must be deterministic — for CFGs, a topological order (or DFS with deterministic tie-breaking) is appropriate.

**What must be preserved** (semantic content):
- Control flow structure (CFG topology)
- Constants, literals, operators
- The *number* and *types* of variables and tables
- Read/write relationships between statements and state

**What must be normalized** (naming):
- Variable names
- Table/map names
- Any compiler-generated identifiers that do not affect semantics (hop IDs, function IDs, instance IDs — though care is needed here; see below)

**Important consideration**: Some identifiers (like hop IDs or function IDs) may carry semantic meaning in certain contexts. The canonicalization logic must be designed with knowledge of which identifiers are purely cosmetic and which affect the verification outcome. This requires careful analysis of the existing codebase — **do not assume all identifiers are safe to normalize**.

### 1.5 Fingerprinting

After canonicalization, the canonical representation is serialized deterministically and hashed (e.g., SHA-256). This hash serves as the cache key.

**Determinism is critical**: The serialization must not depend on HashMap iteration order or any other non-deterministic source. All collections should be sorted before serialization.

### 1.6 File-Based Cache Store

**Storage layout**: A directory-based store where each cache entry is a file, keyed by the fingerprint hash. A sharding scheme (e.g., first 2 hex characters of the hash as a subdirectory) avoids excessive files in a single directory.

**Cache entry contents**: Each entry should record:
- The verification result (verified / failed with counterexample)
- Metadata for invalidation (compiler version, solver version)
- Timestamps and access counts for LRU eviction
- A human-readable summary for debugging

**What to cache and what not to cache**:

| Result | Cache? | Rationale |
|--------|--------|-----------|
| Verified | ✅ Yes | Stable and sound |
| Failed (counterexample) | ✅ Yes | The program is incorrect; re-verification won't change this |
| Timeout | ❌ No | May succeed with more time or with splitting enabled |

**Eviction policy**: When the cache exceeds the configured storage limit, evict entries using LRU (least recently accessed). Entries record an `accessed_at` timestamp updated on each cache hit.

**Invalidation**: If the compiler version or solver version changes between runs, cached results may no longer be valid. The cache should detect version mismatches and invalidate stale entries. The exact granularity of this check (global invalidation vs. per-entry) is a design decision for implementation.

### 1.7 Thread Safety

Since multiple verification threads may access the cache concurrently:
- **Reads** can proceed concurrently (shared lock or lock-free).
- **Writes** should use atomic file operations (write to a temp file, then rename) to avoid partial/corrupt entries.
- An in-memory index (protected by a read-write lock) can accelerate lookups without hitting the filesystem for every query.

### 1.8 Integration Point

The cache check should wrap the existing verification flow. Conceptually:

```
run_cached(unit):
    fingerprint = canonicalize_and_hash(unit)
    if cache.has(fingerprint):
        return cache.get(fingerprint)
    result = run(unit)               // existing verification logic
    if result is definitive:
        cache.store(fingerprint, result)
    return result
```

This is a non-invasive wrapper — the existing `run()` method does not need to change.

### 1.9 CLI Flags

| Flag | Description | Default |
|------|-------------|---------|
| `--cache` | Enable verification result caching | off |
| `--cache-dir <PATH>` | Directory for cache storage | `./<output_dir>/.verification_cache` |
| `--cache-max-size <MB>` | Maximum cache storage | 512 MB |
| `--clear-cache` | Clear all cached results before running | — |
| `--cache-stats` | Print cache hit/miss statistics after the run | off |

### 1.10 Risks and Open Questions

| Item | Notes |
|------|-------|
| **Canonicalization correctness** | The mapping of "which identifiers are cosmetic vs. semantic" requires careful analysis of the codebase. Getting this wrong leads to unsound caching (treating different problems as the same) or unnecessary cache misses (treating the same problem as different). |
| **Serialization determinism** | Must be verified thoroughly — any non-determinism causes phantom cache misses. |
| **Version sensitivity** | How tightly should cache entries be bound to compiler/solver versions? Too tight = frequent invalidation. Too loose = potential unsoundness. |
| **Cache corruption** | What happens if a cache file is corrupted (disk error, killed process mid-write)? The implementation should validate entries on read and discard corrupt ones gracefully. |

## Feature 2: Recursive Commutativity Splitting

### 2.1 Motivation

Some commutativity checks (proving A;B = B;A) are too expensive for the SMT solver to handle within a reasonable time budget. When a check times out, we want to **decompose** it into smaller, independent sub-checks that are each easier to solve.

The key idea: if we split program A into sequential segments A₁;A₂, and we can prove that each segment commutes with B independently (A₁;B = B;A₁ **and** A₂;B = B;A₂), then we can conclude A;B = B;A.

**This approach is sound but incomplete**:
- **Sound**: If all sub-checks pass, the original commutativity holds. (Proof: A;B = A₁;A₂;B = A₁;B;A₂ = B;A₁;A₂ = B;A.)
- **Incomplete**: If a sub-check fails, it does *not* mean the original fails. The sub-checks quantify over all possible intermediate states, but in the original program only specific intermediate states actually occur. A sub-part might not commute with B for *all* states, even though it commutes for the *reachable* states.

### 2.2 Key Requirements

| Requirement | Description |
|-------------|-------------|
| **Timeout-driven** | Splitting only triggers when direct verification exceeds a configurable time bound |
| **CFG-level cuts** | Splits happen on the CFG representation, not on generated Boogie text |
| **Valid cuts only** | Only split at points with clean sequential flow — no branching, no merging, no back-edges crossing the cut |
| **Dataflow-informed** | Use dataflow analysis to choose the best cut point (minimize shared state) |
| **Bounded recursion** | Configurable maximum split depth to prevent runaway decomposition |
| **Observable** | All split decisions, cut point analyses, and sub-problem results are logged |
| **Graceful degradation** | If no valid cut exists, or max depth is reached, return "unknown" rather than failing |

### 2.3 What is a Valid Cut Point?

A valid cut point is a location in the CFG where we can cleanly partition the program into two sequential halves with no control-flow entanglement.

**A CFG node `n` is a valid cut point if ALL of the following hold**:

1. **Single predecessor**: `n` has exactly one incoming edge (no merging of branches at this point)
2. **Single successor**: `n` has exactly one outgoing edge (no branching at this point)
3. **Predecessor is non-branching**: The predecessor of `n` also has exactly one outgoing edge (so the edge into `n` is not one arm of a branch)
4. **Not a loop header**: `n` is not the entry point of a loop
5. **No back-edge crosses**: No loop back-edge has its source at or after `n` and its target at or before `n` (i.e., `n` is not inside a loop body that would be bisected by the cut)
6. **Not entry or exit**: `n` is not the first or last node of the CFG

In summary, a valid cut point sits on a "straight-line" portion of the CFG, with no loops or branches crossing it:

```
VALID:    ─── [a] ──→ [n] ──→ [b] ───     (clean sequential flow)
                        ↑
                    cut here

INVALID:  ─── [a] ──┬→ [n] ──→ [b] ───     (a branches — violates criterion 3)
                     └→ [m] ──→ ...

INVALID:  ─── [a] ──→ [n] ──→ [b] ───     (back-edge from b to a crosses n
                ↑               │            — violates criterion 5)
                └───────────────┘
```

**If no valid cut point exists** in the program being split, we do not split further and return "unknown" for that check.

### 2.4 Choosing the Best Cut Point

When multiple valid cut points exist, we want the "best" one. The primary criterion is to **minimize the state shared across the cut**, because each shared variable becomes a universally quantified variable in the sub-problems (which increases the difficulty for the solver).

**Scoring approach** (to be refined during implementation):

1. **Primary: Minimize live variables across the cut** — Use liveness analysis to determine how many variables are live (defined before the cut and used after the cut). Fewer live variables = less shared state = easier sub-problems.

2. **Secondary: Minimize shared table accesses** — If tables/maps are accessed on both sides of the cut, they contribute to shared state. Tables are typically more expensive for the solver than scalar variables.

3. **Tertiary: Prefer balanced splits** — Among cuts with similar state costs, prefer the one closest to the midpoint of the program. Balanced splits produce sub-problems of similar size, which is generally better for solver performance.

The exact scoring weights should be tunable and may need empirical calibration.

### 2.5 Dataflow Analysis

The splitting feature requires the following dataflow analyses on the CFG:

| Analysis | Direction | Purpose |
|----------|-----------|---------|
| **Liveness** | Backward | Determine which variables are live at each cut point (defines the shared state) |
| **Reaching definitions** | Forward | Understand definition-use chains across cuts |
| **Read/write sets** | Per-node | Determine which tables are accessed on each side of a cut |
| **Back-edge detection** | DFS | Identify loops; ensure cuts don't bisect loop bodies |
| **Loop header identification** | From back-edges | Exclude loop headers as cut points |

Some of these analyses may already exist in the codebase. The implementation should reuse existing infrastructure where possible.

### 2.6 How Splitting Works at the Verification Level

When a direct commutativity check A;B = B;A times out:

1. **Choose which side to split**: Typically the larger program (by some size metric), but this is a heuristic.

2. **Find the best cut point** in that program's CFG (per §2.3–2.5).

3. **Partition the CFG** at the cut point, producing two sub-programs (e.g., A₁ and A₂ where A = A₁;A₂).

4. **Generate sub-commutativity checks**: One check per sub-program vs. the unsplit side.
   - Sub-check 1: A₁;B = B;A₁
   - Sub-check 2: A₂;B = B;A₂

5. **Each sub-check is an independent verification problem**: It havocs (universally quantifies) all relevant state, then runs both orderings and asserts equivalence — exactly like the original check, but with a smaller program fragment.

6. **Recurse**: Each sub-check goes through the same timeout → split cycle. A sub-check may itself be split if it times out and a valid cut exists.

7. **Combine results**:
   - All sub-checks verified → original is **verified**
   - Any sub-check finds a real counterexample → mark as **unknown** for the original (sound: sub-check failure does not imply original failure, due to incompleteness)
   - Any sub-check times out with no further splits possible → **unknown**

### 2.7 Complexity Considerations

**Why splitting helps**: If the solver's runtime is exponential in program size (which is the worst-case behavior for SMT), splitting a size-`n` program into two size-`n/2` pieces can dramatically reduce total time — from `2^n` to `2 × 2^(n/2)`, which is an exponential speedup.

**The overhead**: Each split introduces universally quantified intermediate state. If there are `s` shared variables at the cut, the sub-problems are somewhat larger/harder than they would be without that state. The `s` parameter is why minimizing live variables at the cut point matters.

**Recursive splitting**: At optimal depth, recursive splitting can theoretically convert exponential verification time to polynomial. In practice, the benefit depends on:
- Whether valid cut points with low shared state exist
- Whether the sub-problems are actually provable (incompleteness)
- The structure of the underlying SMT problem

**Key insight**: Splitting helps most when the solver's difficulty comes from path explosion (many branches), and helps least when the difficulty comes from quantifiers or nonlinear arithmetic.

### 2.8 Integration Point

The splitting logic wraps around the existing verification flow:

```
run_with_splitting(unit, timeout, depth):
    if depth > max_depth:
        return UNKNOWN

    result = run(unit, timeout)       // existing verification with time limit
    
    if result is VERIFIED or FAILED:
        return result
    
    if result is TIMEOUT:
        cut_points = find_valid_cut_points(unit.cfg)
        if cut_points is empty:
            return UNKNOWN
        
        best_cut = select_best_cut(cut_points)
        sub_units = partition_at(unit, best_cut)
        
        for sub_unit in sub_units:
            sub_result = run_with_splitting(sub_unit, timeout, depth + 1)
            if sub_result is not VERIFIED:
                return UNKNOWN
        
        return VERIFIED
```

### 2.9 Logging

All split decisions must be observable in compiler logs. At minimum:

- When a timeout triggers a split attempt
- How many valid cut points were found
- Which cut point was selected and why (score breakdown)
- The shared state at the cut (which variables, which tables)
- The result of each sub-problem
- The final combined result and total depth used

This is essential for debugging and for users to understand why verification succeeded, failed, or returned unknown.

### 2.10 CLI Flags

| Flag | Description | Default |
|------|-------------|---------|
| `--split` | Enable timeout-driven commutativity splitting | off |
| `--split-timeout <SECONDS>` | Per-check timeout before triggering a split | 300 |
| `--split-max-depth <N>` | Maximum recursive split depth | 5 |
| `--split-stats` | Enable detailed split decision printing | off |
| `--split-strategy <STRATEGY>` | Cut point selection strategy (`min-state`, `balanced`, `min-state-balanced`) | `min-state-balanced` |

### 2.11 Risks and Open Questions

| Item | Notes |
|------|-------|
| **Incompleteness rate** | How often will splitting produce "unknown" results on programs that are actually commutative? This depends on the problem domain and can only be determined empirically. |
| **CFG partitioning** | The mechanical act of splitting a CFG and producing valid sub-programs requires care. Entry/exit conditions, variable scoping, and state initialization for each sub-program must be handled correctly. |
| **What counts as "program size"?** | The heuristic for deciding which side to split needs a meaningful size metric. Statement count, CFG node count, or some weighted measure? |
| **Interaction with existing analyses** | The splitting feature needs dataflow analysis. How much of this already exists in the codebase, and how much is new? |
| **Timeout granularity** | The split timeout applies to the Boogie/Z3 invocation. Is the existing infrastructure able to enforce time limits on individual verification queries, or does this need new plumbing? |
| **Sub-problem generation** | Generating correct Boogie for sub-problems (with havoc'd intermediate state) requires careful integration with the existing Boogie generation pipeline. |

## Implementation Phasing

These two features are **independent** and can be developed and shipped separately.

### Phase 1: Caching
- Canonicalization and fingerprinting
- File-based cache store with LRU eviction
- CLI flags
- Testing (alpha-equivalence correctness, thread safety, eviction)

### Phase 2: Splitting
- Valid cut point identification on CFGs
- Dataflow analysis for cut point scoring (liveness, read/write sets), already in codebase
- CFG partitioning logic
- Sub-commutativity check generation
- Recursive verification driver with timeout and depth control
- Logging
- CLI flags
- Testing (valid cut detection, split correctness, edge cases)

### Phase 3: Polish
- End-to-end integration testing on real programs
- Performance benchmarking (cache hit rates, split speedups)
- Documentation (user guide for flags, developer guide for extending)
